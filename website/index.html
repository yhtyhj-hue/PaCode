Why PaCode

为「要可控 Agent 基建」的人设计

不只是聊天壳。核心是可检查的确定性基础设施：权限、压缩、工具并发与文件记忆。

  • 默认 MiniMax，国产网关一等公民

    开箱 MiniMax-M3；内置 DeepSeek / 豆包 / GLM / 混元 / 千问 / Kimi 与 Token Plan、Coding Plan 预设。也可用 cc-switch import 从 CC Switch 导入。

  • 配置以 .paude 为准

    运行时以 ~/.paude/ 为源；启动不自动吃 Claude Code 配置。需要时显式 pacode cc-switch import 从 CC Switch / settings 合并进来。

  • 工程级安全门禁

    7 级权限 + tool-gate + Bash 静态分析 + 工作区路径边界。非 TTY 默认 deny，适合脚本与 CI。

  • 可验证的质量车道

    Gate tests + eval harness（含与 Claude Code 的 head-to-head）。改动带回归证据，不是口号。

Performance

与 Claude Code 的实测对照

数据来自仓库 evals/fixtures/m5*/COMPARE.json（2026-07-17）。双方 passRate 均为 100%。墙钟优势部分来自 PaCode 任务并发（默认 3），Claude Code 侧为串行（concurrency=1）——下表如实展示，并标注口径。

模型：双方任务用 claude-sonnet-4-6（经 cc-switch）· Claude Code 2.1.207 · 指标：墙钟 wall / 校验 verify.mjs once-success

Easy suite · 墙钟

PaCode
14.5 s
Claude Code
155.6 s

wall：PaCode 14 524 ms vs CC 155 572 ms · 单任务耗时总和：30.7 s vs 155.6 s · 任务 fix-bug / add-test / small-refactor 均双方通过

Hard suite · 墙钟

PaCode
15.5 s
Claude Code
98.0 s

wall：PaCode 15 469 ms vs CC 97 985 ms · 双方 passRate=1 · multi-file-bug / fail-then-fix / cross-module

怎么读这些数字
  • 测的是「任务能否一次 verify 通过」+ 墙钟，不是 tok/s，也不是 SWE-bench。
  • PaCode 默认并行跑多任务；Claude Code 对照为串行——墙钟差距会放大并发收益。
  • 对照用 sonnet-4-6，与产品默认 MiniMax-M3 不是同一模型；换模型结果会变。
  • 与 OpenCode：仓库尚无共享 M5 fixture，官网不编造延迟数字。

Compare

PaCode · Claude Code · OpenCode

三条产品路线不同：闭源商业 Agent、MIT 社区多模型 Agent、以及以 CC 架构为蓝本的可审计实现。下表帮你选对场景。

维度 PaCode Claude Code OpenCode
许可 BUSL-1.1（个人可用；组织生产需商业授权） 专有（Anthropic 产品） MIT，社区最大
模型默认 MiniMax-M3；国产 Anthropic 兼容 + Token/Coding Plan 预设 Anthropic Claude 家族 75+ Models.dev；OpenAI/Anthropic/本地；另有 Go 订阅
架构取向 对齐 CC：权限层、5 层压缩、工具循环、文件记忆 原版实现，能力面最全 自有 TUI / build·plan 双 Agent，LSP 反馈强
配置与隐私 .paude/ 为准；可显式从 CC Switch 导入 ~/.claude 生态 本地优先；会话可用 SQLite 等本地存储
扩展 Hooks / Skills / Plugins / MCP（含 WS） Hooks / Skills / MCP 等一等公民 MCP + 多端（终端 / 桌面 / IDE）
实测套件 自有 gate + M5 vs CC COMPARE 官方产品；本仓作对照基线 本仓无共享 bench（暂无定性）
适合谁 要 CC 式基建、可控账单、可审计源码的团队/个人 要官方体验与最大能力面、接受 Anthropic 生态 要 MIT、多模型、大社区与桌面端的用户

PaCode 优势

  • 架构对标 Claude Code，权限与压缩路径清晰可测
  • 默认 MiniMax；国产网关 / Token Plan 预设完整
  • 有与 CC 同任务的 passRate / wall 证据链
  • npm 一键装：@sallon/pacode

PaCode 短板

  • Models.dev 不加载专用 SDK 包（Bedrock/Vertex 等），广度仍略逊 OpenCode
  • 核心工具约 31，少于 CC 公开引用的 ~43
  • 无容器级 Bash 沙箱、无公网 Bridge SaaS（有意 Defer）
  • BUSL 对商业组织有授权门槛（相对 MIT 的 OpenCode）
  • 社区体量远小于 OpenCode / 官方 CC

Claude Code

  • 能力面与产品完成度最高
  • 官方支持与模型深度整合
  • 闭源；配置与账号生态绑定 Anthropic
  • 对照跑中串行墙钟更长（本仓 M5 设定）

OpenCode

  • MIT + 超大社区；多模型 / 本地模型自由
  • 终端 + 桌面 + IDE，产品形态丰富
  • 不是「CC 架构复刻」路线，基建细节不同
  • 与本仓无统一 verify 套件，速度不可直接对赌
